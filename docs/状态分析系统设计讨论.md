# 状态分析系统设计讨论总结

## 核心需求

设计一个状态分析系统，支持切换 **全局/当前会话** 的状态选择，包含三种模式：

1. **立即处理模式** - 查看所有消息并选择性回复
2. **提及触发模式** - 一般消息不立即查看，当被提及时才查看缓冲区消息
3. **完全缓存模式** - 所有消息都缓存不处理，等状态变更再处理

## 双LLM架构设计

### 1. 上下文分析LLM (Context Analyzer)
- **职责**: 分析对话上下文
- **运行方式**: 同步/准同步
- **注意点**: 延迟可能影响效果，但作为"提示"而非"引导"作用

### 2. 状态分析LLM (State Analyzer)
- **职责**: 分析当前状态
- **运行方式**: **异步**
- **优势**: 状态分析可以容忍一定延迟

## 关键设计原则

### 提示 vs 引导
- 话题LLM和Planner的输出是**提示信息**，而非**强制引导**
- 真正的主导者是**主LLM**
- 即使LLM请求慢，主模型也只会合理利用这些信息，不会被强行影响

### 延迟处理策略
```
即使话题LLM延迟较高：
├── 发送给话题LLM的上下文永远是最新的
├── 话题LLM会意识到话题转变并释放旧话题
└── 主LLM保持主导权，不会被延迟信息带偏
```

### 实时性权衡
- **话题需要实时性** → 但异步效果损失不大
- **原因**: 不是直接引导LLM输出，而是作为额外信息源
- **条件**: LLM延迟低的话效果可以接受

## 与核心记忆的类比

> "核心记忆也是一股脑丢给LLM，也没有强行联系很多记忆"

- 同样是作为信息源提供给主LLM
- 主LLM自主选择如何利用这些信息
- 不会强制关联或过度依赖

## 实际测试观察

在与回响的对话测试中观察到：
- 基于提示词设计，主模型**合理地利用**话题LLM信息
- **不会强行一直挂在某个话题上**
- 即使LLM请求慢，也不会被完全影响

## 架构优势

1. **解耦** - 状态分析与上下文分析分离
2. **弹性** - 异步处理容忍延迟
3. **可控** - 主LLM保持最终决策权
4. **自适应** - 最新上下文确保话题及时更新
